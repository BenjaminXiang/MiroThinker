# M0.1 + M0.2 — Qwen3-Reranker-8B Client

## Overview

Add a sync HTTP client for the self-hosted Qwen3-Reranker-8B model at `http://100.64.0.27:18006/v1/rerank`. This is the first brick of the Agentic RAG infrastructure (M0 in the 003 execution plan). Without it, M3 RetrievalService cannot produce a final ranked list; M4 chat routes cannot score evidence; M5 web-search refinement cannot re-rank Serper results.

Bundle **M0.2 `load_local_api_key()`** helper into the same PR because it's ~15 lines, used by both the new `RerankerClient` and the existing `EmbeddingClient`, and trying to ship them separately creates an immediate circular dependency.

Output: a working `RerankerClient.rerank(query, documents, top_n=N)` returning a sorted list of `RerankResult(index, score, document)`.

## Problem Frame

The 003 execution plan commits to Qwen3-Reranker-8B as the re-ranking component in the RAG pipeline (see `2026-04-20-003…§0 现状对账`: "Qwen3-Reranker-8B 客户端 — 缺失"). `docs/api.md:179-305` gives a working Python sample that uses `urllib` with proxy bypass. The existing codebase uses `httpx` for all other provider HTTP clients (`vectorizer.py::EmbeddingClient`). The project memory (see `feedback_proxy_llm.md`) mandates `trust_env=False` on all HTTP calls to avoid the 100.64.0.15:7893 proxy poisoning tokens.

We need a client that: (a) follows the project's httpx convention, (b) has proxy isolation baked in, (c) reads API keys via the standard env/file fallback pattern (currently duplicated in `vectorizer.py` and the api.md sample), and (d) is testable without hitting the live endpoint.

## Requirements Trace

- **R1** (from 003 §M0.1): `RerankerClient.rerank(query, documents, top_n=None)` POSTs to `/rerank` and returns results sorted by `relevance_score` desc. (See origin: docs/plans/2026-04-20-003…)
- **R2** (from 003 §M0.1): Empty `documents` returns empty list without making a network call.
- **R3** (from 003 §M0.1): Missing API key raises a clear `RuntimeError` (not a silent empty response).
- **R4** (from 003 §M0.1 + memory `feedback_proxy_llm.md`): Internal httpx client uses `trust_env=False`.
- **R5** (from 003 §M0.1): `top_n > len(documents)` is handled (truncate to `len(documents)`), not raised.
- **R6** (from 003 §M0.2): `load_local_api_key()` reads env in priority `API_KEY > OPENAI_API_KEY > SGLANG_API_KEY`, then `.sglang_api_key` file, then empty string. Never raises.

- **`apps/miroflow-agent/src/data_agents/providers/__init__.py`** — existing exports follow `from .module import Class` + `__all__` pattern. Add `RerankerClient`, `RerankResult` here.

### Institutional Learnings

- `memory/feedback_proxy_llm.md` — Environment proxy at 100.64.0.15:7893 must be cleared before HTTP. `trust_env=False` on httpx.Client achieves this at the client level without poking at env vars.
- `memory/feedback_codex_deviations.md` — Codex has a history of hardcoding generic `api_key` patterns. Brief explicitly: use the new `load_local_api_key()` helper, do NOT inline `os.getenv` in the client class.

### External References

- Serper sample in `docs/api.md` is urllib-based. We deliberately break from it to stay consistent with the rest of the providers directory.

## Key Technical Decisions

- **Sync over async**: First consumer (M3.2 RetrievalService) is itself called from sync FastAPI handlers; async would force an unnecessary event-loop round-trip. If a later async caller appears (e.g., batch indexer), add `AsyncRerankerClient` then.
- **Dataclass for result shape**: `RerankResult(index: int, score: float, document: str)` as `@dataclass(frozen=True, slots=True)` to match the project's contracts.py convention. Avoids a TypedDict dict-access-everywhere ergonomic cost.
- **Fail-loud on missing key, fail-soft on empty docs**: Missing key = misconfiguration (raise `RuntimeError`). Empty documents = legitimate no-candidates case (return `[]` without HTTP call). These map to R3 and R2 respectively.
- **Client ownership of httpx lifecycle**: Accept `client: httpx.Client | None = None`. If None, create one with `trust_env=False`. Support `close()` and context-manager protocol so callers can scope network lifecycle. Mirrors the future M0.3 refactor of `EmbeddingClient`.
- **Helper module name: `local_api_key.py`** (per 003 §M0.2 spec). Not `api_key.py` (too generic). Not `auth.py` (misleading — this isn't auth logic).
- **Helper returns empty string, never raises**: Callers (`RerankerClient`, `EmbeddingClient`) decide whether missing is fatal. R6 makes this explicit.

## Open Questions

### Resolved During Planning

- **Q: Bundle M0.2 helper into this PR or ship separately?** → Bundle. Ship-separately would force `RerankerClient` to inline `os.getenv` for its first landing, then refactor on M0.2. That's churn.
- **Q: Sync or async first?** → Sync. See Key Technical Decisions.
- **Q: Dataclass or dict for `RerankResult`?** → Frozen dataclass with slots, per project convention.
- **Q: Port `EmbeddingClient` to use the new `load_local_api_key()` helper in this PR?** → **No.** Keep diff minimal. Add a TODO comment in `vectorizer.py::EmbeddingClient.__init__` referencing M0.3 where the refactor belongs. Shipping client + helper + refactor of an unrelated class in one PR violates the 003 "one milestone = one PR" rule.

### Deferred to Implementation

- **Exact log message format on HTTP error**: Keep consistent with `EmbeddingClient` — which currently just lets `response.raise_for_status()` propagate without custom wrapping. Mirror that. If a debugging need surfaces at implementation time, add structured logging then.
- **Whether to retry on 5xx**: Not in R1-R6. No retry for M0.1; leave to `RetrievalService` level in M3.2 where retry budget is query-level, not request-level.

## Implementation Units

- [ ] **Unit 1: `load_local_api_key()` helper**

**Goal:** Provide the single canonical way to resolve the local-service API key from env/file, ready for use by `RerankerClient` (this PR) and `EmbeddingClient` (M0.3 refactor).

**Requirements:** R6

**Dependencies:** None.

**Files:**
- Create: `apps/miroflow-agent/src/data_agents/providers/local_api_key.py`
- Test: `apps/miroflow-agent/tests/providers/__init__.py` (empty package file)
- Test: `apps/miroflow-agent/tests/providers/test_local_api_key.py`

**Approach:**
- Module-private `_ENV_NAMES = ("API_KEY", "OPENAI_API_KEY", "SGLANG_API_KEY")` constant; iterate in order.
- `Path.cwd()` is not reliable as "repo root". Accept optional `repo_root: Path | None = None`; when None, derive from `__file__` walking up to locate `.sglang_api_key`. If not found, return empty string (never raise).
- Strip whitespace from env/file values. Empty-after-strip is treated as unset.

**Execution note:** Test-first. Write the 3 helper tests first, see them fail, implement.

**Patterns to follow:**
- The api.md sample at `docs/api.md:67-76` shows the intended env precedence.
- `apps/miroflow-agent/src/data_agents/professor/vectorizer.py:29` uses `api_key = ""` default with identical semantics.

**Test scenarios:**
- Happy path — `API_KEY=abc` set → returns `"abc"`
- Happy path — no env set, file `.sglang_api_key` contains `xyz\n` → returns `"xyz"` (trimmed)
- Edge case — `API_KEY=" "` (whitespace only) + `OPENAI_API_KEY=real` → returns `"real"` (env precedence skips empty)
- Edge case — nothing set, no file → returns `""` (not None, not a raise)
- Integration — monkeypatch all three env vars to distinct values, assert `API_KEY` wins (priority order)

**Verification:**
- All scenarios pass via `uv run pytest apps/miroflow-agent/tests/providers/test_local_api_key.py`
- `grep -n "os.getenv.*API_KEY" apps/miroflow-agent/src/data_agents/providers/local_api_key.py` shows exactly one occurrence per env name (no surprise reads elsewhere)

- [ ] **Unit 2: `RerankerClient` + `RerankResult`**

**Goal:** Sync client that POSTs to the Qwen3-Reranker endpoint and returns a sorted `list[RerankResult]`.

**Requirements:** R1, R2, R3, R4, R5

**Dependencies:** Unit 1 (imports `load_local_api_key`).

**Files:**
- Create: `apps/miroflow-agent/src/data_agents/providers/rerank.py`
- Test: `apps/miroflow-agent/tests/providers/test_rerank.py`
- Modify: `apps/miroflow-agent/src/data_agents/providers/__init__.py` (export `RerankerClient`, `RerankResult`)

**Approach:**
- Module constants mirroring `vectorizer.py`:
  - `_DEFAULT_RERANK_URL = "http://100.64.0.27:18006/v1"`
  - `_DEFAULT_MODEL = "qwen3-reranker-8b"`
  - `_DEFAULT_TIMEOUT = 60.0`
- `RerankResult` as `@dataclass(frozen=True, slots=True)` with fields `index: int`, `score: float`, `document: str`.
- `RerankerClient.__init__(*, base_url=_DEFAULT_RERANK_URL, api_key: str | None = None, model=_DEFAULT_MODEL, timeout=_DEFAULT_TIMEOUT, client: httpx.Client | None = None)`.
  - If `api_key is None`, fall back to `load_local_api_key()`. If still empty, defer the raise until `rerank()` is actually called (matches `web_search.py` lazy-raise style).
  - If `client is None`, construct `httpx.Client(trust_env=False, timeout=timeout)` and track ownership so `close()` / `__exit__` dispose it. If caller provided one, do NOT close it.
- `rerank(query, documents, *, top_n: int | None = None) -> list[RerankResult]`:
  - If `documents` is empty: return `[]` immediately (no HTTP call).
  - If api_key empty: `raise RuntimeError("RerankerClient: no API key. Set API_KEY env or .sglang_api_key file.")`.
  - Effective `top_n` = `min(top_n, len(documents))` if `top_n` is set; else `len(documents)`.
  - POST body: `{"model": self.model, "query": query, "documents": list(documents), "top_n": effective_top_n}`.
  - Header: `Authorization: Bearer <key>` + `Content-Type: application/json`.
  - Parse `response.json()["results"]`. Each entry has `index`, `relevance_score`, and optionally `document` (dict with `text`, or string, or missing — handle per api.md:284-295 precedent: fall back to `documents[index]` when absent).
  - Return results already sorted by server; do NOT re-sort (server is authoritative). But add a post-parse assertion (logger.warning, not raise) if the sort breaks — guards against silent API behavior change.
- Context manager: `__enter__ → self`, `__exit__ → self.close()`. `close()` only closes when we own the client.

**Execution note:** Test-first. Write the 6 client tests first (4 from 003 + 2 new — top_n truncation and HTTP error propagation), see them fail, implement.

**Technical design:**

```
RerankerClient.rerank(query, documents, top_n)
│
├─ if len(documents) == 0 → return []
├─ if self.api_key == "" → raise RuntimeError
├─ effective_top_n = min(top_n or len(documents), len(documents))
│
├─ POST {base_url}/rerank
│   body: {model, query, documents, top_n: effective_top_n}
│   headers: Authorization, Content-Type
│   (internal httpx.Client has trust_env=False)
│
├─ response.raise_for_status()  # propagate HTTP errors
│
└─ parse response.json()["results"] →
    for each entry:
       RerankResult(
         index = entry["index"],
         score = entry["relevance_score"],
         document = entry.get("document", {}).get("text")
                    or (isinstance(entry["document"], str) and entry["document"])
                    or documents[entry["index"]]
       )
    return sorted-by-server list (warn on non-desc sort)
```

*Directional guidance for reviewers — not implementation specification.*

**Patterns to follow:**
- `apps/miroflow-agent/src/data_agents/professor/vectorizer.py:28-60` — httpx POST + `raise_for_status` + sorted parse.
- `apps/miroflow-agent/src/data_agents/providers/web_search.py:32-34` — `trust_env = False` idiom.

**Test scenarios:**
- Happy path — mock httpx returns 3 results with scores `[0.9, 0.5, 0.1]`, assert returned list is 3 items, `index` + `score` + `document` populated, order matches server.
- Happy path — call with `top_n=2`, assert we send `"top_n": 2` in request body (use `respx` or MagicMock capture).
- Edge case — `rerank("q", [], top_n=3)` returns `[]` and does not invoke httpx (spy on `httpx.Client.post` / injected mock).
- Edge case — `rerank("q", ["a", "b"], top_n=10)` sends `"top_n": 2` (truncated to `len(documents)`).
- Edge case — response entry has `document` as a plain string vs. `{"text": "..."}` vs. missing — all three parse correctly (falls back to `documents[index]`).
- Error path — `RerankerClient(api_key="").rerank("q", ["a"])` raises `RuntimeError` with message containing "API key".
- Error path — httpx returns 500 → `rerank()` propagates `httpx.HTTPStatusError` (or equivalent); caller sees it, we don't swallow.
- Integration — context manager usage: `with RerankerClient() as client: ...` — after the `with` block, `client._http.is_closed` is True when client was internally owned; False when the user injected their own.
- Integration — `trust_env=False` is actually set on the internally-created httpx.Client (introspect the client instance, assert `.trust_env is False`).

**Verification:**
- All scenarios pass via `uv run pytest apps/miroflow-agent/tests/providers/test_rerank.py`
- `python -c "from src.data_agents.providers import RerankerClient, RerankResult; print(RerankerClient)"` works from `apps/miroflow-agent/`
- `ruff check apps/miroflow-agent/src/data_agents/providers/rerank.py apps/miroflow-agent/tests/providers/test_rerank.py` clean
- No new non-test files import `requests` or `urllib` in this diff — httpx only

## System-Wide Impact

- **Interaction graph:** First consumer is M3.2 `RetrievalService` (not yet written); this PR only exposes the client. No existing production code path reaches `RerankerClient` in this milestone.
- **Error propagation:** HTTP errors propagate via httpx exception types to the caller. `RetrievalService` (M3.2) will decide retry/fallback policy at the query level, not here.
- **State lifecycle risks:** httpx.Client instance lifecycle. If caller forgets to close an internally-owned client, Python GC will eventually close it, but file descriptors may linger in long-running batch scripts. Mitigation: context manager support in Unit 2.
- **API surface parity:** `EmbeddingClient` will be refactored in M0.3 to use `load_local_api_key()`. Planned, not done here. The helper ships with both callers in mind.
- **Unchanged invariants:** `EmbeddingClient` public API does not change in this PR. `web_search.py::WebSearchProvider` untouched.

## Risks & Dependencies

| Risk | Mitigation |
|------|------------|
| Qwen3-Reranker response schema differs from `docs/api.md:284-295` (e.g., `document` field nested differently) | Test scenario explicitly covers all 3 shapes (dict-with-text, plain string, missing). Implementation mirrors api.md sample's `get → isinstance → fallback to documents[index]` cascade. |
| Local endpoint `100.64.0.27:18006` unreachable in CI | Tests use mocked httpx (`respx` or unittest.mock), not live endpoint. No CI dependency on the server. |
| Codex hardcodes `api_key = os.getenv(...)` when writing impl (memory: `feedback_codex_deviations.md`) | Cross-validation step in CLAUDE.md Stage 4 — Claude Code reads the generated file and verifies `RerankerClient` imports and calls `load_local_api_key()`, not inlined env reads. |
| Proxy env leaks from parent shell despite `trust_env=False` (vectorizer.py has the same exposure) | Integration test asserts `client.trust_env is False` on the internally-owned client. Can manually test with `export http_proxy=http://bad.host:9999 && uv run pytest -k reranker` to confirm the client ignores it. |
| `response.json()` returns unexpected non-dict / missing `"results"` key | Mock an unhappy-path response (`{}`); assert a clear error. Plan: `KeyError: "results"` propagates, caller sees it. Not worth wrapping for M0.1. |
