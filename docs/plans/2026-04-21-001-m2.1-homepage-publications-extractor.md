# M2.1 — Homepage Publications Extractor

## Overview

Build a pure function that takes a professor homepage's HTML + page URL and returns a structured list of `HomepagePublication` records. Downstream (M2.2) will resolve each record's title against OpenAlex/arxiv/Serper to get full metadata; M2.3 fetches full text; M2.4 orchestrates.

No network calls in this unit. No database writes. Just HTML → structured list.

This is the **first brick** in the pipeline-inversion direction that the 001 audit called out as the single highest-leverage fix for the 47% paper-reject rate: prof homepage publications are authoritative and should flow through to the canonical paper store without LLM adjudication. M2.1 builds only the extraction primitive; the write path lands in M2.4.

## Problem Frame

Professors list their representative publications on their school homepages, usually under a "Publications" / "论文" / "Selected Works" heading. The current `homepage_crawler.py` detects that such a section exists and scores its signal strength, but never extracts individual items. Every downstream step (`paper_collector.py`, `paper_identity_gate.py`) relies on Semantic Scholar / OpenAlex author discovery, which misses ~47% of Chinese professors due to name-variant + same-name collisions.

If we can parse the homepage list itself — even imperfectly — every downstream step has a vastly stronger ground truth to anchor on.

Challenge: five Shenzhen school families, five different HTML patterns. Within each school, individual faculty pages also diverge (some use `<ol>`, some `<ul>`, some `<p>` paragraphs, some `<table>`). The extractor must be format-agnostic without becoming a pile of school-specific hacks.

## Requirements Trace

- **R1** (from 003 §M2.1): Function signature `extract_publications_from_html(html: str, *, page_url: str) -> list[HomepagePublication]`.
- **R2** (from 003 §M2.1): `HomepagePublication` fields: `raw_title`, `clean_title`, `authors_text`, `venue_text`, `year`, `source_url`, `source_anchor`.
- **R3** (from 003 §M2.1): Happy-path returns ≥20 items on a real homepage with ~20-item publications list.
- **R4** (from 003 §M2.1): `clean_title` length is reasonable (10-300 characters); strip common prefixes (`[1]`, `1.`, `(1)`) and suffixes (`[J]`, `, 2023.`, trailing author names).
- **R5** (from 003 §M2.1): At least one of `authors_text` or `venue_text` is non-empty per item (otherwise downstream M2.2 can't disambiguate).
- **R6** (derived): No HTTP calls, no DB writes. Pure function. Deterministic on identical HTML.
- **R7** (derived from 003 §M2.1 5-school test matrix): Extract from ol/ul/p/table layouts; tolerate year-anchored paragraphs (`"2023. Title. Venue."`).

## Scope Boundaries

**In scope:**
- Pure function in `homepage_publications.py`
- `HomepagePublication` frozen dataclass
- Title cleaning heuristics (prefix/suffix strip, whitespace normalize)
- Publications-section detection (reuse `homepage_crawler.py` heuristics, don't duplicate)
- Item extraction from ol/ul/p/table/dl structures
- Year extraction from item text
- Best-effort `authors_text` and `venue_text` split (don't over-invest — M2.2 re-parses against OpenAlex)
- Dedup within a single homepage (same normalized title + year)
- Unit tests with 5 synthetic HTML archetypes + 1 per-school real snippet

**Out of scope (deferred):**
- Title → OpenAlex/arxiv resolution (M2.2)
- PDF full-text fetching (M2.3)
- Multi-page pagination (main homepages don't paginate; defer to M2.4 orchestrator if needed)
- JavaScript-rendered publications (headless browser) — defer until a real prof is found where this matters
- Author name filtering (`NameVariants` type is deferred in M1.1; extractor must be designed to accept an optional filter hook but skip using it for now)
- Cross-language duplicate detection (CN/EN titles of same paper — let M2.2 sort it out via OpenAlex)
- Writing to Postgres / `professor_paper_link` (M2.4)
- Confidence score per item (M2.4 will assign one at link time)

## Key Technical Decisions

- **Pure function, not a class.** State-free, deterministic, trivially testable. Matches `title_cleaner` pattern.
- **Format-agnostic extraction, not per-school adapters.** Five heuristics cascade: (H) ol/ul item, (P) paragraph under Publications heading, (T) table rows, (Y) year-anchored paragraph, (D) definition list. Each returns the same `HomepagePublication` shape. The adapter-architecture learning says to keep these universal.
- **Reuse `homepage_crawler.py` section detection.** Factor the existing heading-keyword list into a shared constant (`_PUBLICATIONS_HEADING_RE`) accessible from both. Don't fork the regex.
- **Best-effort authors/venue split; let M2.2 clean up.** Use simple heuristics (split on `"."` after title, look for common venue keywords like "Journal", "Conference", "Proceedings"). Title quality matters most; authors/venue are bonus.
- **Dedup key: `normalize(clean_title) + year`.** `normalize` = lowercase + whitespace collapse + punctuation strip. Prefer the first occurrence (document order).
- **Fixture strategy: hybrid (synthetic archetypes + 1 real sample per school).** Synthetic HTML covers the 5 format archetypes for TDD hermeticity. A single real-world sample per school (checked into `tests/.../fixtures/homepage/` with source attribution) provides smoke-test realism without committing us to maintaining copies of 783 faculty pages.
- **`source_anchor` captures DOI/arxiv `<a href>` when the homepage links directly.** This is the single most valuable unstructured signal. Downstream M2.2 will prefer this over re-resolving the title when available.
- **Optional author-filter hook, don't wire it.** Signature: `extract_publications_from_html(..., author_filter: Callable[[str], bool] | None = None)`. If None, accept all items. M1 can wire a `NameVariants`-backed filter later without changing the signature.
- **Log, don't raise, on parse quality issues.** Use the existing project `logger`. If fewer than 3 items extracted from a page that clearly has a publications section, log a warning so the M2.4 orchestrator can flag it to `pipeline_issue`. Never raise from the extractor — an empty list is a valid result.

## Open Questions

### Resolved During Planning

- **Q: Pure function or class?** → Pure function. See Key Technical Decisions.
- **Q: Synthetic vs real HTML fixtures?** → Hybrid. See Key Technical Decisions.
- **Q: Accept an `author_filter` parameter now or later?** → Accept the parameter now with default `None`; don't implement the filtering. Keeps the signature stable for M1.
- **Q: How aggressive should title cleaning be?** → Conservative. Strip obvious prefixes (`[1]`, `1.`, `(1)`) and obvious trailing markers (`[J]`, `[C]`, `[J/OL]`). Preserve punctuation inside the title (colons, hyphens). Downstream M2.2 normalizes more aggressively when querying OpenAlex.
- **Q: Min-title-length filter?** → Yes, 10 characters. Anything shorter is almost certainly a prefix/stub that survived cleaning.
- **Q: Max-items per homepage?** → 200 hard cap. If a page has more, truncate and log. Real faculty pages top out around 100; 200 guards against listing-page false positives.
- **Q: What if the HTML has multiple Publications sections (e.g., "Selected Publications" + "Full List")?** → Extract from all detected sections, dedup at the end. Don't try to pick "the right" section.

### Deferred to Implementation

- **Exact regex for prefix/suffix stripping.** Start with obvious patterns, extend as fixture tests surface new ones. The eval set (M1.4 style) doesn't yet exist for this; implementer iterates during RED→GREEN.
- **Year extraction ambiguity when multiple years appear.** Paragraph might contain "2023 Proceedings of 2022 ICML". Implementer picks by heuristic at implement time (prefer trailing year vs leading year). Tests will pin the behavior.
- **Exact `venue_text` heuristic.** Needs to be tuned against real fixture content; don't over-design before seeing data.

## High-Level Technical Design

> *This illustrates the intended approach and is directional guidance for review, not implementation specification. The implementing agent should treat it as context, not code to reproduce.*

```
extract_publications_from_html(html, *, page_url, author_filter=None)
│
├── soup = BeautifulSoup(html, "lxml")
├── strip nav/footer/header/aside
│
├── sections = find_publications_sections(soup)
│     Strategies tried in order (first non-empty wins per block):
│       1. <h*> / <div class="*publication*"> with heading matching _PUBLICATIONS_HEADING_RE
│       2. <section id|class containing "publication" | "论文" | "papers">
│       3. Any block whose heading text matches the pubs regex
│
├── for each section:
│     candidates = extract_items_from_section(section, page_url)
│       Strategies tried in order:
│         (H) direct <ol>/<ul> children → one item per <li>
│         (P) consecutive <p> siblings → one item per <p>
│         (T) <table> rows → one item per <tr> (title = largest text cell)
│         (Y) year-heading groups → items under each <h4>/<strong>"2023"</strong>
│         (D) <dl> structure → items from <dt>/<dd>
│
├── for each candidate:
│     parse_item_text(raw_text, page_url) →
│       - strip prefix ([1], 1., (1))
│       - strip suffix ([J], [C], [J/OL], trailing year-only)
│       - extract year (regex 19XX|20XX, validate ≤ current_year + 1)
│       - split into title / authors / venue by heuristic (period-delimited cascade)
│       - find <a href> within candidate: if matches DOI/arxiv → source_anchor
│       - if clean_title length < 10 → drop
│       - if author_filter is not None and not author_filter(authors_text) → drop
│
├── dedup by (normalize(clean_title), year)
├── cap at 200 items; log if exceeded
└── return list[HomepagePublication]
```

This design stays format-agnostic: each strategy is a self-contained path that either produces candidates or falls through. No school-specific code.

*Integration/realism tests:*
- Integration — Faculty page with a `<a href="https://doi.org/...">` inside a list item → `source_anchor` is the DOI URL.
- Integration — Faculty page with arxiv URL in anchor (`arxiv.org/abs/2310.12345`) → `source_anchor` captures it.
- Integration — Single HTML with both "Selected Publications" and "Full Publications List" sections → merged, deduped.
- Integration — Real snippet from 清华 SIGS format (committed to `fixtures/homepage/real_tsinghua_sigs_sample.html`) → extracts ≥ 5 items (small fixture); mostly a smoke test, not strict assertions.

*Edge cases:*
- Edge case — Empty HTML → returns `[]`, no raise.
- Edge case — HTML with no publications section → returns `[]`.
- Edge case — HTML with pubs section but 0 items (empty `<ul>`) → returns `[]`, logs warning.
- Edge case — Duplicate item in the same section (same title, same year) → deduped; 1 record returned.
- Edge case — Duplicate item across two sections (Selected + Full) → 1 record returned.
- Edge case — Item with no extractable year (e.g., "to appear" paragraph) → record created, `year=None`.
- Edge case — Item with title < 10 chars after cleaning (e.g., "TBD." or "1.") → dropped, not returned.
- Edge case — HTML with 250 items → capped at 200, warning logged.

*Error paths:*
- Error path — Malformed HTML (unclosed tags, nested `<li>` soup) → lxml recovers, extraction proceeds on best-effort basis, no raise.
- Error path — Non-UTF-8 content already decoded as `str` with replacement chars → normalizes via `clean_paper_title`, still returns extracted items.

*Contract tests:*
- Contract — `author_filter=None` default yields all items.
- Contract — `author_filter=lambda text: "Smith" in (text or "")` drops items whose `authors_text` doesn't contain "Smith".
- Contract — Function is pure: same HTML → same output across two calls (order, contents, dedup all deterministic).

## System-Wide Impact

- **Interaction graph:** Net-new module. First caller will be M2.4 orchestrator (`run_homepage_paper_ingest.py`). No existing code calls it in this PR.
- **Error propagation:** Extractor never raises on content issues. Only raises on programmer errors (invalid arg types). M2.4 wraps calls and files `pipeline_issue` rows for quality-warning cases.
- **State lifecycle risks:** None — pure function, no persistent state.
- **API surface parity:** The `author_filter` hook is the only piece designed for future evolution. Its signature (`Callable[[str], bool] | None`) matches how `NameVariants` + its matcher function (M1) will plug in.
- **Integration coverage:** Covered by Unit 3 smoke tests against real HTML.
- **Unchanged invariants:** `homepage_crawler.py` scoring logic and signal detection are not modified beyond factoring out the shared publications-heading regex. No existing callers of `homepage_crawler` should see behavior changes. Verified by running the existing `test_homepage_crawler.py` suite post-refactor.

## Risks & Dependencies

| Risk | Mitigation |
|------|------------|
| Real-world HTML doesn't match any of the 5 archetypes (e.g., JavaScript-rendered SPA faculty page) | Unit 3 smoke tests catch it immediately; expand archetypes or flag as headless-browser requirement for M2.4 follow-up. |
| Title-cleaning heuristics too aggressive, drop legit items | Min-title-length (10 chars) + eval set grown from real fixture failures during Unit 3. Log-don't-drop when near the threshold so M2.4 can inspect. |
| Title-cleaning heuristics too lenient, keep garbage | `authors_text` + `venue_text` being empty is a quality signal for M2.2 to deprioritize. Not this unit's problem to solve. |
| `homepage_crawler.py` shared regex refactor breaks existing callers | Run full `test_homepage_crawler.py` suite before and after the refactor; diff should be zero test deltas. |
| Fixture drift (a school changes its homepage HTML template) | Real fixtures are checked in as static snapshots; not re-fetched on test runs. Drift is a manual refresh, not a flaky test. |
| Extracting from multiple publications sections double-counts or over-dedups | Unit 2 edge case `Duplicate item across two sections` pins the behavior. |
| Year extraction picks the wrong year when multiple appear | Tests pin the heuristic; implementer documents the choice in a one-line comment near the function. |
| Hard 200-item cap truncates legit extremely-productive faculty (rare but possible) | Warning logged → M2.4 reviewer catches via `pipeline_issue`. 200 >> observed max (~100) so the cap is defensive only. |
| Privacy / licensing concern with committing real homepage HTML snippets | Each fixture is minimal (publications block only), attributed to its public URL, and obviously already-public academic information. If a concern surfaces, swap to fully-synthetic + skip Unit 3; still keeps Unit 2 green. |
