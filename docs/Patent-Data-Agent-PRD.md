# 专利数据采集智能体 — 产品需求文档

> 本文档定义专利域的特有需求。通用架构、MiroThinker 实现映射、质量维度、更新发布规则见 [共享技术规范](./Data-Agent-Shared-Spec.md)。术语定义见 [术语表](./index.md#术语表)。

## 一、为什么需要独立的专利数据采集智能体

专利数据不是论文或企业数据的简单附属字段。它有自己独立的数据来源、字段体系、去重规则和用户理解门槛。

专利域需要同时解决：

- 从平台导出 xlsx 稳定导入
- 专利条目标准化
- 用户可读的通俗解释
- 企业 / 教授 / 发明人关联
- 支持专利检索、统计和趋势分析

因此，需要一个独立的 `Patent-Data-Agent` 负责专利数据的导入、清洗、摘要生成、关联和发布。

---

## 二、核心目标

### 2.1 一句话目标

从平台导出的专利 xlsx 出发，构建可检索、可解释、可关联的本地专利知识库，为专利查询、企业画像、教授关联和技术趋势分析提供支撑。

### 2.2 核心成功标准

专利域必须支持：

- 按申请人检索专利
- 按技术主题语义检索专利
- 按专利号精确查询
- 企业 / 教授关联跳转
- 专利解释性回答

实现方式见 [共享技术规范 §3](./Data-Agent-Shared-Spec.md#三与当前-mirothinker-实现的映射)。

## 三、数据来源与范围

### 3.1 主数据来源

专利域主数据来源为：

- 平台导出的专利 `xlsx`

当前已确认的样例文件为：

- [`2025-12-05 专利.xlsx`](./2025-12-05%20专利.xlsx)

### 3.2 第一阶段范围

第一阶段规则应为：

- 全量导入导出的专利数据

不在入库前做过窄裁剪，例如：

- 只导机器人专利
- 只导已关联企业的专利
- 只导某个 IPC 小类

筛选应主要发生在：

- 检索层
- 服务层
- 分析层

### 3.3 辅助来源

辅助来源包括：

- 公开专利网页
- 企业官网或新闻稿
- Web Search

这些来源主要用于：

- 补充解释
- 补充关联证据
- 验证异常记录

不用于替代 xlsx 主骨架。

## 四、数据模型与对外契约

### 4.1 最低发布字段

发布层至少应包含：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `id` | 是 | 稳定主键，建议 `PAT-*` |
| `title` | 是 | 专利标题 |
| `title_en` | 否 | 英文标题 |
| `patent_number` | 否 | 专利号 / 公开号 |
| `applicants` | 是 | 申请人列表 |
| `inventors` | 否 | 发明人列表 |
| `patent_type` | 否 | 发明 / 实用新型 / 外观设计 |
| `filing_date` | 否 | 申请日期 |
| `publication_date` | 否 | 公开日期 |
| `grant_date` | 否 | 授权日期 |
| `abstract` | 否 | 原始摘要 |
| `summary_text` | 是 | 用户可读摘要 |
| `technology_effect` | 否 | 技术效果 / 技术要点 |
| `ipc_codes` | 否 | IPC 分类 |
| `company_ids` | 否 | 关联企业 ID 列表 |
| `professor_ids` | 否 | 关联教授 ID 列表 |
| `evidence` | 是 | 来源列表 |
| `last_updated` | 是 | 最后更新时间 |

> `patent_number` 格式参考：中国专利通常为 `CN` + 9-10 位数字 + 1 位字母（如 `CN117873146A`）。导入时应标准化为统一格式。

### 4.2 用户可读摘要字段

专利域必须预生成用户可读摘要字段，至少包括：

- `summary_text`

这个字段的目标不是把专利文献简单翻译成白话，而是帮助用户快速理解：

- 这项专利在做什么
- 解决了什么问题
- 技术路线大致是什么
- 对企业 / 教授有什么意义

如条件允许，可进一步拆成结构化摘要：

- `what`
- `problem`
- `method`
- `effect`

但一期最小要求是稳定的 `summary_text`。

### 4.3 存储要求

专利域独立 PostgreSQL 库 + Milvus collection。详见 [共享技术规范 §6](./Data-Agent-Shared-Spec.md#六物理存储与向量化建议)。

## 五、采集与清洗流程

### 5.1 总体流程

```text
专利 xlsx
  -> 表头识别与原始解析
  -> 字段标准化
  -> 去重与归并
  -> 生成 summary_text
  -> 建立 company_ids / professor_ids
  -> 发布到 patent domain PostgreSQL + Milvus
```

### 5.2 xlsx 导入

导入阶段至少完成：

1. 读取专利 xlsx
2. 识别真实表头
3. 抽取标题、摘要、申请人、发明人、日期、类型等字段
4. 生成统一记录结构
5. 输出导入报告

### 5.3 标准化

至少标准化以下字段：

- 专利标题
- 申请人名称
- 发明人名称
- 专利类型
- 日期字段
- IPC 分类号

### 5.4 去重与归并

专利去重优先级建议：

1. 专利号 / 公开号
2. 标题高相似度 + 申请人高重叠

归并原则：

- 保留最完整的一条记录
- 合并能互补的字段

### 5.5 摘要生成

建议分工：

- LLM 负责：
  - `summary_text`
  - 技术效果和路线的提炼
- Python / 离线脚本负责：
  - 日期和编号格式规整
  - IPC 解析
  - 名称标准化

### 5.6 辅助补充

当 xlsx 字段不足或存在明显异常时，可用：

- Web Search
- 公开专利网页
- 企业公开信息

做辅助补充和交叉验证。

## 六、关联规则

### 6.1 企业关联

`company_ids` 建立优先参考：

1. 申请人标准化名称
2. 企业域已发布对象
3. 公开网页证据

### 6.2 教授关联

`professor_ids` 建立优先参考：

1. 发明人姓名
2. 所属机构
3. 论文 / 企业 /公开网页中的交叉证据

### 6.3 关联写入原则

写入关联字段时：

- 置信度足够才落正式关联
- 不确定时可先记录候选关系，待验证后回填

## 七、质量保证

通用质量维度和验证流程见 [共享技术规范 §7](./Data-Agent-Shared-Spec.md#七数据质量与验证)。

### 7.1 专利域特有校验

1. `title` 不得缺失
2. `applicants` 不得缺失
3. `summary_text` 不得缺失
4. 日期字段格式合法
5. `patent_number` 若存在，应符合 `CN` + 数字 + 字母格式
6. `company_ids` / `professor_ids` 存在时，关联应可追溯

### 7.2 重点验证对象

- 申请人名称歧义大的专利
- 发明人重名的专利
- 解释性摘要质量差的专利
- 重要企业或重点教授关联的专利

## 九、更新策略

- 主更新节奏：月度
- 每次以最新导出 xlsx 为主重新导入
- 已有记录按主键或稳定去重规则更新
- 辅助 Web Search 只做按需补证，不做主流程

## 十、验收标准

| 指标 | 要求 | 测试集 | 样本量 | 评判标准 |
| --- | --- | --- | --- | --- |
| 导入覆盖率 | ≥ 当期导出 xlsx 有效记录的 95% | 专利导出 xlsx | 全量 | 自动化比对 |
| `summary_text` 完整率 | ≥ 90% | 全量专利 | 全量 | 自动化校验 |
| 企业关联准确率 | ≥ 90% | 专利-企业关联标注集 | ≥ 100 条 | 人工判定 |
| 教授关联准确率 | ≥ 85% | 专利-教授关联标注集 | ≥ 50 条 | 人工判定 |
| 去重准确率 | ≥ 95% | 含已知重复对的标注集 | ≥ 100 对 | 人工判定 |
| 检索效果 | Top-5 相关率 ≥ 85% | Agentic-RAG 测试集中专利类 query | ≥ 50 条 | 人工评估相关性 |
