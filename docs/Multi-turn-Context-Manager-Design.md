# 多轮对话上下文管理器 — 技术设计文档

> 本设计服务于 [Agentic-RAG-PRD](./Agentic-RAG-PRD.md) 中定义的多轮对话能力（§4.2）。
> 数据模型和字段定义见 [共享技术规范](./Data-Agent-Shared-Spec.md) 和 [术语表](./index.md#术语表)。

## 一、为什么需要独立的上下文管理器

用户在与科创检索智能体交互时，超过 50% 的查询是多轮追问（如"看看他的论文""这些企业中哪些融过资"）。系统必须在对话轮次之间维护上下文，否则每次查询都是孤立的——用户不得不反复输入完整的实体名称和筛选条件。

核心挑战不在"记住上一轮说了什么"，而在三个更深层的问题：

**第一，指代消解。** 用户说"他""这家公司""上面那篇论文"时，系统需要将模糊指代映射到具体的实体（教授、企业、论文、专利）。这需要理解指代类型（人称代词、指示代词、省略主语）和候选实体列表。

**第二，跨模块跳转。** 用户先查了一位教授（命中教授库），然后问"他有哪些专利"（跳转专利库）。上下文管理器需要传递实体信息到目标模块，同时保持对话连贯性。

**第三，话题切换检测。** 用户可能在对话中途突然换话题（"对了，深圳做协作机器人的企业有哪些"），此时需要清除旧上下文，避免新查询被旧实体污染。

## 二、上下文模型

### 2.1 会话状态定义

```
SessionContext {
  session_id: string              # 会话唯一标识
  user_id: string                 # 用户标识（微信 OpenID 或 Web 端会话 ID）
  created_at: timestamp           # 会话创建时间
  last_active_at: timestamp       # 最后活跃时间（30 分钟无活动则过期）
  ttl_seconds: int = 1800         # 会话有效期，默认 30 分钟

  turns: ConversationTurn[]       # 对话历史（最近 N 轮）
  active_entities: EntityStack    # 当前活跃实体栈
  current_module: string|null     # 当前命中的模块（professor/company/paper/patent）
  last_result_set: ResultRef|null # 上一轮检索结果集引用
}

ConversationTurn {
  turn_index: int
  user_input: string
  intent_type: string             # A/B/C/D/E/F/G（查询类型）
  target_modules: string[]        # 本次查询命中的模块
  entities_mentioned: Entity[]    # 本轮显式提到的实体
  result_summary: string          # 结果摘要（用于后续指代消解）
  timestamp: timestamp
}

Entity {
  type: enum(professor, company, paper, patent)
  id: string                      # 数据库中的实体 ID
  display_name: string            # 展示名称（如"张三教授""优必选"）
  source_turn: int                # 首次出现的轮次
  confidence: float               # 实体识别置信度
}

EntityStack {
  # 按类型维护最近出现的实体，每类最多保留 3 个
  professors: Entity[]    # 最多 3 个
  companies: Entity[]     # 最多 3 个
  papers: Entity[]        # 最多 3 个
  patents: Entity[]       # 最多 3 个
  # 栈顶 = 最近提到的实体，指代消解优先使用栈顶
}
```

### 2.2 实体栈的行为规则

| 操作 | 触发条件 | 行为 |
| --- | --- | --- |
| Push | 检索返回明确匹配的实体 | 压入对应类型的栈顶，若栈满则淘汰栈底 |
| Pop | 用户明确切换话题 | 清空所有栈 |
| Peek | 指代消解时 | 从对应类型栈顶取候选实体 |
| Cross-peek | 跨模块跳转时 | 从源模块类型栈顶取实体，传递给目标模块 |

## 三、指代消解引擎

### 3.1 指代类型分类

| 指代类型 | 示例 | 消解策略 |
| --- | --- | --- |
| 人称代词 | "他""她""他的" | 从 `professors` 栈顶取实体 |
| 指示代词（单数） | "这家公司""这篇论文""这个专利" | 从对应类型栈顶取实体 |
| 指示代词（复数） | "这些企业""上述教授" | 从对应类型栈取全部实体 |
| 省略主语 | "有什么专利" → 补全为"他/她有什么专利" | 结合上一轮意图推断主语 |
| 属性追问 | "融过什么资""发了什么论文" | 推断目标类型 → 从对应栈取实体 |
| 比较追问 | "和 XX 比呢" | 保留上一轮实体 + 新实体做比较 |

### 3.2 消解流程

```
用户输入: "他有什么专利"
  │
  ├─ 1. 意图识别 → 目标模块: patent, 查询类型: 发明人查询
  │
  ├─ 2. 指代检测 → 发现 "他" (人称代词)
  │
  ├─ 3. 候选实体 → professors 栈顶 → Entity(type=professor, id="PROF-xxx", name="张三")
  │
  ├─ 4. 置信度判断:
  │    ├─ 栈顶实体置信度 ≥ 0.8 → 直接使用
  │    ├─ 栈顶实体置信度 0.5-0.8 → 追问确认 ("您是指张三教授吗？")
  │    └─ 栈顶实体置信度 < 0.5 或栈为空 → 请求用户明确输入
  │
  └─ 5. 组装查询 → search_patents(filters={inventor: "张三"}, query="张三 专利")
```

### 3.3 歧义场景处理

| 场景 | 处理方式 |
| --- | --- |
| 上一轮返回多个教授，本轮说"他的论文" | 列出候选让用户选择，不猜测 |
| 上一轮查了企业，本轮说"他的专利" | "他"通常指人 → 检查 professors 栈是否非空；若空则追问"您是指哪家企业？" |
| 用户说"上面那个"但已过 3 轮 | 降低置信度，超过 5 轮则强制要求用户明确输入 |
| 栈中有同类型多个实体 | 默认使用栈顶（最近提到的），若用户意图不明确则列出候选 |

## 四、跨模块上下文传递

### 4.1 跳转路径矩阵

基于 Agentic-RAG-PRD 定义的 8 条跨模块查询路径：

```
          ┌──────────┐
          │  教授库   │
          └──┬───┬──┘
             │   │
    ┌────────┘   └────────┐
    ▼                     ▼
┌──────────┐         ┌──────────┐
│  论文库   │         │  企业库   │
└────┬─────┘         └────┬─────┘
     │                     │
     └──────┐   ┌──────────┘
            ▼   ▼
        ┌──────────┐
        │  专利库   │
        └──────────┘
```

### 4.2 跳转上下文传递规则

| 跳转路径 | 传递什么 | 查询组装 |
| --- | --- | --- |
| 教授 → 论文 | 教授 ID + 姓名 | `search_papers(filters={professor_id: id})` |
| 教授 → 专利 | 教授姓名 | `search_patents(filters={inventor: name})` |
| 教授 → 企业 | 教授 ID | `get_professor_companies(id)` → 关联查询 |
| 企业 → 专利 | 企业名称 | `search_patents(filters={applicant: name})` |
| 企业 → 教授 | 企业 ID | 反向查询 `company_roles` 中包含该企业的教授 |
| 论文 → 教授 | 论文作者列表 | 逐一匹配 `professors` 表 |
| 专利 → 企业 | 专利申请人 | `search_companies(query=applicant)` |
| 专利 → 教授 | 专利发明人列表 | 逐一匹配 `professors` 表 |

### 4.3 跳转时的上下文增强

跳转不仅仅是传递实体 ID，还需要传递上下文信息帮助目标模块更精准地检索：

```
跳转上下文 = {
  source_entity: Entity,           # 源实体（触发跳转的实体）
  source_module: string,           # 源模块
  target_module: string,           # 目标模块
  additional_filters: dict,        # 用户在追问中附加的筛选条件
  original_query_context: string,  # 原始查询的语义摘要（帮助目标模块理解意图）
}
```

示例：用户先查"优必选"，然后问"他们有哪些关于人形机器人的专利"
- `source_entity`: `{type: company, name: "优必选"}`
- `target_module`: patent
- `additional_filters`: `{keywords: ["人形机器人"]}`
- `original_query_context`: "用户想了解优必选在人形机器人方向的专利布局"

## 五、话题切换检测

### 5.1 切换信号

| 信号 | 示例 | 置信度 |
| --- | --- | --- |
| 新实体 + 新模块 | "介绍一下南科大的李四"（之前在查企业） | 高（0.9+） |
| 显式切换词 | "换个话题""对了""另外" | 高（0.9+） |
| 完整的独立查询 | "深圳有哪些做具身智能的教授"（完整句子，无指代） | 中（0.7+） |
| 模糊信号 | "还有什么"（可能是追问也可能是新话题） | 低（< 0.5）→ 优先按追问处理 |

### 5.2 切换决策流程

```
用户输入
  │
  ├─ 是否包含显式切换词？──是──→ 清空实体栈，按新查询处理
  │
  ├─ 是否包含指代词？──是──→ 按追问处理，不切换
  │
  ├─ 是否是完整独立查询（无指代、包含新实体名）？
  │    ├─ 是 → 检查新实体是否与栈中实体相关
  │    │    ├─ 相关 → 按追问处理（如"介绍一下丁文伯的合作者"）
  │    │    └─ 不相关 → 清空实体栈，按新查询处理
  │    └─ 否 → 按追问处理
  │
  └─ 无法判断 → 按追问处理（保守策略，宁可多保持上下文也不要误清）
```

## 六、与 MiroThinker Orchestrator 的集成

### 6.1 架构定位

多轮上下文管理器是 RAG 智能体的中间层，位于用户接口和四个检索模块之间：

```
用户 (微信 H5 / Web 端)
  │
  ▼
┌─────────────────────────────────┐
│         对话管理接口层            │
│  ┌───────────────────────────┐  │
│  │   Multi-turn Context      │  │
│  │   Manager                 │  │
│  │  ┌─────────┐ ┌─────────┐  │  │
│  │  │ Session │ │ Entity  │  │  │
│  │  │ State   │ │ Stack   │  │  │
│  │  └─────────┘ └─────────┘  │  │
│  │  ┌─────────┐ ┌─────────┐  │  │
│  │  │ Coref   │ │ Topic   │  │  │
│  │  │ Resolver│ │ Detector│  │  │
│  │  └─────────┘ └─────────┘  │  │
│  └───────────────────────────┘  │
│              │                   │
│    ┌─────────┼─────────┐        │
│    ▼         ▼         ▼        │
│ ┌──────┐ ┌──────┐ ┌──────┐     │
│ │ 教授 │ │ 企业 │ │ 论文 │ ... │
│ │ 模块 │ │ 模块 │ │ 模块 │     │
│ └──────┘ └──────┘ └──────┘     │
└─────────────────────────────────┘
```

### 6.2 借鉴 MiroThinker 的模式

MiroThinker Orchestrator 的多轮推理模式可以直接借鉴：

| MiroThinker 概念 | 上下文管理器对应 |
| --- | --- |
| Orchestrator 多轮循环 | Session 管理 + turn 迭代 |
| 上下文压缩 | Entity Stack（只保留关键实体，不做全文压缩） |
| 子 Agent 委托 | 跨模块跳转（将实体上下文传递给目标模块） |
| 终止条件 | 会话 TTL（30 分钟） |
| 工具调用失败回滚 | 指代消解失败时的追问澄清 |

### 6.3 会话存储

```
存储方案: Redis
  Key: session:{session_id}
  Value: SessionContext (JSON)
  TTL: 1800 秒 (30 分钟)
  过期策略: 每次用户交互刷新 TTL

降级方案: 内存存储（单机部署时）
  风险: 服务重启丢失会话
  缓解: 会话丢失后用户只需重新输入实体名，体验降级但不报错
```

## 七、LLM Prompt 设计

### 7.1 意图识别 + 指代消解 Prompt

上下文管理器的核心判断由 LLM 完成，通过一个合并的 prompt 同时完成意图识别和指代消解：

```
你是对话上下文分析器。给定用户的当前输入和对话历史，输出 JSON:

{
  "intent_type": "A|B|C|D|E|F|G",
  "target_modules": ["professor", "company", "paper", "patent"],
  "coreferences": [
    {"mention": "他", "resolved_entity": {"type": "professor", "id": "PROF-xxx", "name": "张三"}, "confidence": 0.95}
  ],
  "is_topic_switch": false,
  "additional_filters": {},
  "explicit_entities": [
    {"type": "company", "name": "优必选", "is_new": true}
  ]
}

对话历史:
{最近 5 轮对话摘要}

当前用户输入: {用户输入}

当前活跃实体:
- 教授栈: [{name: "张三", id: "PROF-xxx"}, {name: "李四", id: "PROF-yyy"}]
- 企业栈: [{name: "优必选", id: "COMP-aaa"}]
- ...

规则:
1. 指代消解: 将 "他""这家公司" 等指代表达映射到实体栈中的具体实体
2. 置信度: 0.9+ 直接使用, 0.5-0.9 需追问确认, <0.5 请求明确输入
3. 话题切换: 包含全新实体名 + 新模块类型时判定为切换
4. 保守策略: 无法判断时按追问处理
5. 禁止猜测: 不确定的指代不要强行匹配
```

### 7.2 实体提取 Prompt

从用户输入和检索结果中提取实体，压入实体栈：

```
从以下文本中提取科创领域实体:
- 教授名称（中文姓名 + "教授"/"老师"等称谓）
- 企业名称（公司全称或简称）
- 论文标题
- 专利号（CN 开头的编号格式）

输出 JSON 数组: [{type, name, confidence}]
```

## 八、性能与 SLA

| 指标 | 要求 | 说明 |
| --- | --- | --- |
| 上下文加载 | < 50ms | 从 Redis 读取 SessionContext |
| 指代消解延迟 | < 500ms | LLM 调用（可与意图识别合并，不增加额外延迟） |
| 实体栈操作 | < 10ms | 纯内存操作 |
| 会话存储容量 | 单用户 ≤ 10KB | Entity Stack 限制每类 3 个，turn 历史限制最近 10 轮 |
| 并发会话数 | 1000+ | Redis 天然支持 |

## 九、验收标准

| 测试场景 | 验收标准 |
| --- | --- |
| "查一下南科大张三" → "他的论文" | 正确消解"他"为张三，返回论文列表 |
| "优必选" → "他们的专利" → "发明人是深圳教授吗" | 三轮跨模块跳转，每步实体传递正确 |
| "介绍一下丁文伯" → "深圳有哪些做具身智能的教授" | 话题切换检测正确，新查询不被旧实体污染 |
| "看看他的论文"（会话为空） | 请求用户明确输入，不返回错误 |
| "这家公司呢"（上一轮返回 3 家企业） | 列出候选让用户选择，不猜测 |
| 会话超过 30 分钟无活动 | 会话过期，新查询按全新上下文处理 |
