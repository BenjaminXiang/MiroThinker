# V2.0-深圳科创检索增强智能体 — 需求文档


## 一、产品目标与边界

### 1.1 一句话目标

为深圳科创生态的参与者（政府管理人员、产业研究员、投资机构、高校科研人员、创业团队）提供一个**对话式的科创信息检索入口**——用户用自然语言提问，系统根据查询意图**智能路由到教授、企业、论文、专利四个数据库中的一个或多个**，结合 Web Search 实时补充，返回结构化、可追溯的回答。大多数查询只需命中单个数据库，少数全景式查询才需要跨库聚合。

### 1.2 核心用户场景

| 谁在用 | 什么场景 | 期望得到什么 | 查询类型 |
| --- | --- | --- | --- |
| 科创委管理人员 | “深圳做具身智能的教授和企业有哪些” | 一张全景图：哪些学者在做、哪些企业在做、有多少专利 | D：跨库聚合 |
| 科创委管理人员 | “介绍清华的丁文伯” → “他参与创立了哪些企业” | 先看教授画像，再看关联企业 | A → C：单库查询 → 跨库跳转 |
| 产业研究员 | “深圳有哪些具身智能灵巧手厂商” → “他们在数据层面分别是什么路线” | 厂商列表 + 各家技术路线对比 | B：单库语义检索 + 多轮收窄 |
| 产业研究员 | “具身智能合成数据发展方向上有几种实现方法，代表厂商有哪些” | 技术路线综述 + 代表厂商 | E：科创知识问答 |
| 投资人 | “介绍无界智航的相关信息” | 企业画像（需先消歧同名企业） | G → A：消歧 → 单库查询 |
| 投资人 | “优必选有哪些专利” → “专利 CN117873146A 的详细信息” | 专利列表 → 单条专利详情 | A → A：单库精确查询 |
| 高校教授 | “深圳有哪些做力控的企业可以合作” | 按技术方向匹配的企业列表 | A：单库语义检索 |
| 创业团队 | “我想找 PCB 打板，有哪些推荐” → “上述哪些是深圳的” | 供应商推荐 → 按地域筛选 | B：单库检索 + Web Search + 多轮收窄 |

### 1.3 本期不做什么

| 不做的事 | 理由 |
| --- | --- |
| 不做通用问答（闲聊、天气、翻译等） | 聚焦科创信息检索，不是通用 AI 助手。对于完全无关的问题（如“深圳旅游哪里不能去”），礼貌拒答并引导回科创话题 |
| 不做深度行业研究报告生成 | 科创领域知识问答（类型 E）仅提供简明综述 + 数据佐证，不生成万字级研究报告 |
| 不做用户注册登录体系 | 嵌入微信公众号，依赖微信身份 |
| 不做数据可视化大屏 | 一期聚焦对话式交互，图表仅在后端管理中出现 |
| 不做教授/企业之间的关系图谱 | 有价值但不是一期核心，可从数据中后续推导 |

> **边界说明：科创领域知识问答（类型 E）的处理原则**
>
> 实际使用中，用户会问一些与科创相关但本地四库无法直接回答的知识性问题（如“具身智能合成数据有几种技术路线”）。这类问题不应简单拒答，而应通过 Web Search + LLM 知识生成来回答，但需明确标注“综合自网络搜索和 AI 分析，非本地数据库结果”。这是系统作为“科创信息入口”的增值能力，但不是核心功能——核心功能仍然是四库的结构化检索。

## 二、系统整体行为描述

用户在微信公众号中打开智能体，输入一句自然语言，系统需要完成以下事情：

**判断查询类型** → **路由到正确的检索策略** → **执行检索（可能只查一个库，也可能跨库或走 Web Search）** → **把结果组织成人能看懂的回答** → **标注数据来源**

具体来说：

1. **意图识别与查询分类**：首先判断用户的问题属于哪种查询类型（详见 2.1 节）。不是所有问题都需要查数据库——有些只需单库精确查询，有些需要 Web Search 补充，有些则应礼貌拒答。如果判断不了，追问用户。

2. **语义路由**：根据查询类型，把请求分发到对应的检索策略。单库查询直接路由到目标模块；跨库查询按优先级依次执行；超出范围的问题直接由 LLM 生成拒答引导。

3. **检索执行**：在对应数据库中执行语义检索（理解含义，不只是关键词匹配）+ 结构化筛选（按职称、注册资本、年份等条件过滤）。本地库结果不足时，按策略触发 Web Search 补充。

4. **结果组织**：用大语言模型将检索结果组织为自然语言回答 + 结构化卡片，附上数据来源标注。

5. **多轮对话**：支持上下文。用户说“看看他的论文”，系统知道“他”是上一轮提到的教授。用户在多轮追问中可能自然地从一个库跳到另一个库（如先查教授，再问“他有什么专利”），这属于**对话驱动的跨库跳转**，不是联合检索。

### 2.1 查询类型与检索策略

从实际测试问题中归纳，用户查询可分为以下六种类型，每种类型对应不同的检索策略：

#### 类型 A：单库精确查询（占比最高，约 50%）

用户意图明确指向单个数据库，系统只需在一个库中检索。

| 代表性问题 | 命中数据库 | 检索策略 |
| --- | --- | --- |
| “介绍清华的丁文伯” | 教授库 | 按姓名 + 院校精确匹配，返回教授画像 |
| “优必选有哪些专利” | 专利库 | 按申请人检索，返回专利列表 + 统计 |
| “XX 公司的技术实力怎么样” | 企业库 | 按企业名称检索，返回企业画像 |
| “专利 CN117873146A 的详细信息是什么” | 专利库 | 按专利号精确查询，返回专利详情 |
| “王 XX 教授最近发了什么论文” | 论文库 | 按作者检索，返回论文列表 |

**检索策略**：意图识别 → 路由到单个模块 → 执行语义检索 + 结构化筛选 → 返回结果。

#### 类型 B：单库语义检索 + 多轮收窄（约 20%）

用户先用一个宽泛的语义查询，然后通过追问逐步收窄范围。全程只在同一个库中操作。

| 代表性问题（多轮） | 命中数据库 | 检索策略 |
| --- | --- | --- |
| ① “中国成熟的酒店送餐机器人供应商” → ② “上述哪些在深圳” → ③ “能按电梯的有哪些” | 企业库 | 第一轮语义检索（可能触发 Web Search 补充）→ 后续轮在结果集上追加筛选条件 |
| ① “深圳有哪些做具身智能的灵巧手厂商” → ② “他们在数据层面分别是什么路线” | 企业库 | 第一轮语义检索 → 第二轮对结果集做深度信息提取 |
| ① “我想找 PCB 打板，有哪些推荐” → ② “上述哪些是深圳的企业” | 企业库 | 第一轮语义检索 + Web Search → 第二轮按地域筛选 |

**检索策略**：首轮宽检索 → 维护结果集上下文 → 后续轮在结果集上叠加条件或提取深度信息。注意第一轮可能超出本地库范围（如“全中国”），需要 Web Search 兜底。

#### 类型 C：对话驱动的跨库跳转（约 15%）

用户在多轮对话中自然地从一个库跳到另一个库。这不是“联合检索”，而是**上下文携带的串行跳转**。

| 代表性问题（多轮） | 跳转路径 | 检索策略 |
| --- | --- | --- |
| ① “介绍清华的丁文伯” → ② “他参与创立了哪些企业” | 教授库 → 企业库 | 第一轮查教授库 → 第二轮携带教授姓名跳转企业库检索 |
| ① “XX 公司的专利布局” → ② “这些专利的发明人有哪些是深圳高校教授” | 企业库 → 专利库 → 教授库 | 串行跳转，每步结果作为下一步的输入 |

**检索策略**：维护对话上下文中的实体（人名、公司名、专利号等）→ 识别跳转意图 → 携带上下文实体到目标库检索。

#### 类型 D：全景式跨库聚合（约 5%，真正的“联合检索”）

极少数查询需要同时从多个库中拉取数据并聚合展示。这是唯一需要并行多库检索的场景。

| 代表性问题 | 命中数据库 | 检索策略 |
| --- | --- | --- |
| “深圳做具身智能的教授和企业有哪些” | 教授库 + 企业库 | 同一关键词并行检索两个库，结果分区展示 |
| “协作机器人领域近两年的论文趋势和专利布局” | 论文库 + 专利库 | 并行检索 + 交叉分析 |

**检索策略**：意图识别发现多个目标实体类型 → 并行分发到多个模块 → 各模块独立检索 → LLM 聚合为统一回答。

#### 类型 E：科创领域知识问答（约 5%）

用户问的是科创领域的知识性问题，本地四个库中没有直接答案，但问题与科创主题相关，应当回答。

| 代表性问题 | 命中数据库 | 检索策略 |
| --- | --- | --- |
| “在具身智能的合成数据发展方向上，有几种实现方法，代表厂商有哪些” | 无直接命中 | Web Search + LLM 知识生成 |
| “在具身智能的运动和操作层面，数据需求有什么不同” | 无直接命中 | LLM 知识生成，可引用本地库中相关企业/论文作为佐证 |
| “模拟器生成数据路线中有哪些具体方式” | 无直接命中 | Web Search + LLM 知识生成 |

**检索策略**：识别为知识问答 → 优先尝试从本地库中找相关线索（如相关企业的技术描述、相关论文摘要）→ 触发 Web Search 补充 → LLM 综合生成回答，明确标注“综合自网络搜索和 AI 分析”。

#### 类型 F：超出范围的查询（应拒答）

与深圳科创完全无关的问题，系统应礼貌拒答并引导用户回到科创话题。

| 代表性问题 | 处理方式 |
| --- | --- |
| “深圳旅游有哪些涉及黄赌毒的地方不能去” | 礼貌拒答：“我是深圳科创信息助手，这个问题超出了我的服务范围。我可以帮您查找深圳的科技企业、高校教授、论文和专利信息，请问有什么科创相关的问题吗？” |
| “帮我写一首诗” | 同上 |
| “今天天气怎么样” | 同上 |

#### 类型 G：存在歧义需澄清的查询

用户查询的实体名称存在多个匹配结果（如同名企业、同名教授），系统应主动列出候选项让用户选择。

| 代表性问题 | 处理方式 |
| --- | --- |
| “介绍无界智航的相关信息”（存在“深圳无界智航科技”和“广南县无界智航科技”两家公司） | 列出候选企业，询问用户：“找到两家名称相近的企业，请问您想了解哪一家？” |
| “查一下张三教授”（多所高校有同名教授） | 列出候选教授及其院校，让用户确认 |

**检索策略**：检索结果存在多个高置信度匹配 → 不猜测，主动展示候选列表 → 用户确认后返回详情。

### 2.2 检索策略决策流程

```plaintext
用户输入
  │
  ├─ 是否与科创相关？──否──→ 类型 F：礼貌拒答
  │
  是
  │
  ├─ 能否明确命中某个数据库？
  │    │
  │    ├─ 命中单个库 ──→ 类型 A / B：单库检索
  │    │
  │    ├─ 命中多个库 ──→ 类型 D：并行多库聚合
  │    │
  │    └─ 无法命中任何库 ──→ 类型 E：知识问答（Web Search + LLM）
  │
  ├─ 检索结果是否存在歧义？──是──→ 类型 G：追问澄清
  │
  └─ 是否为多轮追问中的跨库跳转？──是──→ 类型 C：携带上下文跳转
```

## 三、五大功能模块需求

---

### 模块一：查找教授

**S（背景）：** 深圳汇聚了南科大、深大、港中深、哈工深等多所高校的科研人才。这些教授的信息散落在各高校官网，格式不统一、更新不及时。科创委和产业界需要一个统一入口来查找和了解这些教授。

**C（约束）：**

- 数据来源：本地教授数据库（由“导师信息抓取清洗智能体”产出的结构化数据）为主，Web Search 实时补充为辅

- 教授范围：深圳本地高校全量教师（含在深有实体研究机构的外地高校，如清华深研院、北大深研院）

- 本地库检索结果不足时（< 3 条），或用户查询包含“最新”“最近”等时效性词汇时，自动触发 Web Search 补充

- Web Search 结果需经 LLM 提取结构化信息后才能展示，并明确标注“来自网络搜索”

- 所有回答必须标注数据来源和最后更新时间

**A（需要实现的能力）：**

1. **自然语言教授搜索**

   - 按姓名精确搜索：“查一下南科大的张三教授”

   - 按研究方向语义搜索：“深圳有哪些做计算机视觉的教授”

   - 按条件组合搜索：“港中深做 NLP 方向的副教授以上”

   - 模糊口语化查询：“谁在深圳搞机器人比较厉害”

2. **教授画像卡片展示**

   - 基本信息：姓名、职称、院系、高校

   - 研究方向：标签 + 自然语言描述

   - 学术指标：H-index、总引用数、近 5 年论文数

   - 代表成果：Top-3 论文（标题 + 年份 + 引用数）

   - 联系方式：邮箱、个人主页链接

   - 来源标注：本地库 / Web Search

3. **教授关联查询**

   - “看看他的论文” → 跳转论文模块

   - “他有什么专利” → 跳转专利模块

   - “他和哪些企业有合作” → 跳转企业模块（如有数据）

**F（验收标准）：**

- 输入“深圳有哪些做具身智能的教授”，返回结果中包含南科大、深大等相关教授，Top-5 结果相关率 ≥ 85%

- 输入一个本地库中不存在的教授姓名，系统触发 Web Search 补充，结果标注“来自网络搜索”

- 教授画像卡片中所有字段有值（必填字段），数据来源和更新时间清晰标注

- 意图识别准确率 ≥ 90%（100 条教授相关测试 query）

---

### 模块二：查找企业

**S（背景）：** 深圳科创企业数量庞大且持续增长。目前已有超过 6,000 家企业数据（每月递增），数据分为两部分：企名片 API 提供的工商基础数据，以及乙方清洗的高价值数据（含产品描述、PR 稿件等）。需要将这两部分数据融合，支持语义化检索——用户搜“做手术机器人的公司”，不仅匹配关键词，还能召回“微创手术辅助系统”“术中导航设备”等语义相关的企业。

**C（约束）：**

- 数据来源一：企名片 API（工商注册、股东、融资、法律风险等结构化数据），实时/日更

- 数据来源二：乙方清洗数据（产品描述、技术栈、PR 稿件、行业标签），月度批量导入，格式为 Excel/CSV

- 企业数据每月增长，系统必须支持高效的增量导入流程

- 两个数据源通过统一社会信用代码关联融合

- 企名片 API 有调用成本，需做缓存（建议 7 天）

**A（需要实现的能力）：**

1. **语义化企业检索**

   - 语义搜索：理解查询含义，不仅匹配关键词。搜索范围覆盖产品描述、PR 稿件、行业标签、技术栈

   - 结构化筛选叠加：在语义检索基础上可叠加注册资本、成立年限、融资轮次等条件

   - 排序：默认按语义相关度，可切换为按注册资本、成立时间、专利数量

2. **企业信息卡片展示**

   - 基本工商信息：企业名称、法人、注册资本、成立日期

   - 业务概述：产品/服务描述（LLM 摘要为 2-3 句话）

   - 技术标签：行业标签 + 技术栈标签

   - 融资信息：最新融资轮次、金额、投资方

   - 知识产权：专利数量（关联专利模块）

   - 相关 PR：最近 3 篇 PR 稿件标题（可展开查看）

3. **企业数据月度增量导入**

   - 支持 Excel/CSV 格式上传

   - 自动去重（基于统一社会信用代码）：已存在则更新，不存在则新增

   - 导入完成后生成报告：新增 N 条、更新 N 条、失败 N 条及原因

4. **企名片数据实时融合**

   - 用户查看企业详情时，实时调用企名片 API 获取最新工商数据

   - 企名片数据与乙方清洗数据合并展示为完整企业画像

**F（验收标准）：**

- 输入“深圳做协作机器人的企业有哪些，融过 A 轮以上的”，返回结果语义相关且融资条件筛选正确

- 上传一份包含 100 条企业数据的 Excel，导入成功率 ≥ 95%，导入报告准确

- 企业卡片中同时展示企名片数据和乙方清洗数据，信息融合无矛盾

- 6,000+ 企业库中的语义检索响应时间 ≤ 5 秒

### 模块三：查找论文

**S（背景）：** 前沿科研论文几乎全部为英文，发表在 Arxiv 等平台上。科创委的用户群体以中文为主，直接阅读英文论文门槛高。需要一个系统来抓取深圳教授相关的论文，提供中文翻译和智能摘要，让非学术背景的用户也能快速了解最新研究动态。

**C（约束）：**

- 数据来源：Arxiv API 为主（按深圳教授名单 + 科创委关注领域关键词抓取）

- 更新频率：每日增量抓取过去 24 小时新论文；每月全量校验

- 论文需与教授模块关联：论文作者如果是已收录的深圳教授，需建立关联

- 翻译和摘要在入库时预计算完成，用户查询时直接返回，不实时生成

- 只处理中文和英文

**A（需要实现的能力）：**

1. **论文语义检索**

   - 自然语言查询：“最近关于大模型在机器人规划中应用的论文”

   - 按教授查询：“王 XX 教授最近发了什么论文”

   - 按时间范围筛选：“2025 年以来的具身智能论文”

   - 按 Arxiv 分类筛选：[cs. RO](http://cs.RO)（机器人学）、[cs. CV](http://cs.CV)（计算机视觉）等

2. **论文翻译与智能摘要**

   - 直译层：论文标题和摘要的高质量中文翻译，保留学术术语准确性

   - 智能摘要层：LLM 阅读论文后生成约 200 字中文解读——这篇论文做了什么、为什么重要、核心创新点是什么。比直译摘要更通俗易懂

3. **论文结果展示**

   - 中英文标题（可切换）

   - 中文智能摘要（默认）+ 英文原始摘要（可展开）

   - 作者列表（深圳教授高亮标记，可点击跳转教授画像）

   - 发表日期、Arxiv 分类、引用数

   - PDF 下载链接

   - “相关论文”推荐（基于语义相似度）

4. **论文数据自动更新**

   - 每日自动抓取 Arxiv 新论文

   - 自动去重（基于 DOI 和标题相似度）

   - 自动生成中文翻译和智能摘要

   - 自动关联深圳教授

**F（验收标准）：**

- 输入“最近有什么关于人形机器人运动控制的新论文”，返回近期相关论文，Top-5 相关率 ≥ 85%

- 论文的中文智能摘要通俗易懂，非学术背景用户能理解核心内容（人工评估 ≥ 4.0/5.0）

- 深圳教授发表的论文在作者列表中高亮标记，点击可跳转教授画像

- 论文数据每日自动更新，无需人工干预

### 模块四：查找专利

**S（背景）：** IDEA 已采购了深圳地区机器人及相关产业的专利数据（万级规模，每月更新）。传统专利检索需要了解 IPC 分类号和布尔检索语法，门槛很高。需要让用户用日常语言就能完成专利检索，并提供专利分析能力。

**C（约束）：**

- 数据来源：IDEA 采购的专利原始数据，月度批量导入

- 专利范围：深圳地区机器人及相关产业，万级规模且每月增长

- 每条专利需要 LLM 生成通俗解读（将专业的权利要求书翻译为普通人能看懂的语言）

- 专利需与企业模块和教授模块关联：申请人如果是已收录企业则关联，发明人如果是已收录教授则关联

**A（需要实现的能力）：**

1. **自然语言专利查询**

   - 技术主题查询：“深圳有哪些关于柔性夹爪的专利”

   - 申请人查询：“XX 公司申请了多少专利”

   - 发明人查询：“王 XX 教授有哪些专利”

   - 组合查询：“2024 年以来深圳企业申请的人形机器人相关发明专利”

   - 趋势查询：“深圳机器人专利近三年的申请趋势”

2. **专利信息展示**

   - 专利名称、专利号、类型标签（发明/实用新型/外观设计）

   - 状态标签（已授权 / 申请中 / 已失效）

   - 申请人（如为已收录企业则可跳转企业画像）

   - 发明人列表（如为已收录教授则高亮并可跳转）

   - 摘要原文 + LLM 通俗解读

   - 申请日期、公开日期、授权日期

   - IPC 分类号（附中文释义）

3. **专利分析功能**

   - 申请人排行榜：深圳机器人领域专利数量 Top-N 企业/机构

   - 技术领域分布：按 IPC 分类统计专利分布

   - 时间趋势分析：按年/月统计专利申请量变化趋势

4. **专利数据月度更新**

   - IDEA 提供原始数据文件，通过后台批量导入

   - 自动去重（基于专利号）、清洗标准化、LLM 生成通俗解读

   - 每次更新后生成报告（新增 N 条、更新 N 条）

**F（验收标准）：**

- 输入“深圳做四足机器人的企业都有哪些专利布局”，返回专利统计 + 申请人排行 + 技术方向分布

- 专利的 LLM 通俗解读让非专业用户能理解该专利在做什么（人工评估 ≥ 4.0/5.0）

- 申请人为已收录企业时，可点击跳转企业画像

- 月度数据导入后，新专利可被检索到，导入报告准确

### 模块五：后端管理

**S（背景）：** 系统上线后，科创委内部管理人员和运维人员需要了解智能体的使用情况、发现和跟进用户遇到的问题、管理各模块的数据。这个模块不面向普通用户，只供内部管理使用。

**C（约束）：**

- 仅限授权管理员访问

- 不需要精美 UI，功能可用即可

- 问题清单需支持从用户反馈、系统自动检测、低评分三个渠道收集

**A（需要实现的能力）：**

1. **使用情况仪表盘**

   - 总体指标：日/周/月活跃用户数、总查询次数、平均会话轮次

   - 模块使用分布：各模块（教授/企业/论文/专利）的查询占比

   - 热门查询：Top-20 高频查询关键词/主题

   - 查询质量：意图识别准确率、用户好评率、无结果查询占比

   - 响应性能：平均响应时间、P95 响应时间

   - 数据规模：各数据库当前记录数、本月新增数

2. **问题清单管理**

   - 问题来源：用户主动反馈（对话中点“反馈问题”）、系统自动识别（意图识别失败、检索无结果、连续追问等异常模式）、低评分触发

   - 每条问题记录：问题类型、来源、关联会话（可回溯完整对话）、关联模块、优先级、状态（待处理/处理中/已解决）、处理人、处理记录

3. **数据管理**

   - 教授数据：新增/编辑/删除、批量导入

   - 企业数据：月度增量上传、导入执行、结果查看

   - 专利数据：月度导入、处理进度监控、更新报告

   - 论文数据：爬虫运行状态、数据管道监控、异常告警

4. **系统配置**

   - 意图识别阈值调整

   - Web Search 触发条件配置

   - 检索结果数量上限配置

   - 缓存策略配置

**F（验收标准）：**

- 仪表盘数据与实际系统数据一致，刷新后实时更新

- 用户在对话中点击“反馈问题”后，问题清单中出现对应记录，可回溯完整对话上下文

- 通过后台上传企业/专利数据文件，导入流程正常完成，导入报告准确

- 非授权用户无法访问后端管理页面

## 四、跨模块能力需求

### 4.1 跨模块关联查询

四个业务模块不是孤立的，它们之间存在天然的关联关系。系统需要支持用户在一次对话中自然地跨模块查询：

| 查询路径 | 用户怎么问 |
| --- | --- |
| 教授 → 论文 | “王 XX 教授最近发了什么论文” |
| 教授 → 专利 | “李 XX 教授有哪些专利” |
| 教授 → 企业 | “张 XX 教授和哪些企业有合作” |
| 企业 → 专利 | “XX 公司的专利布局是什么” |
| 企业 → 教授 | “XX 公司的技术顾问有谁” |
| 论文 → 教授 | “这篇论文的作者在深圳哪个高校” |
| 专利 → 企业 | “这个专利的申请人是哪家公司” |
| 专利 → 教授 | “这个专利的发明人是哪位教授” |

### 4.2 多轮对话与上下文

- 支持上下文维护：用户先查了一位教授，然后说“看看他的论文”，系统知道“他”是谁

- 支持指代消解：“他”“这家公司”“上面那篇论文”等指代表达

- 支持上下文切换：用户明确换话题时，清除旧上下文

- 单次会话上下文保持 30 分钟

### 4.3 数据来源标注与可追溯性

所有回答必须标注数据来源，包括：数据来自哪个数据库、最后更新时间、是否包含 Web Search 实时补充。当本地数据与 Web Search 数据冲突时，优先展示本地数据并标注差异。

## 五、技术指标与 SLA（合同约束）

> 以下指标依据《国先中心-太昊信息-智能体委托开发合同》第二条制定。

### 5.1 响应速度

| 指标 | 要求 | 说明 |
| --- | --- | --- |
| 首字生成时间（TTFT） | 5 - 25 秒 | 常规查询场景。通过流式输出降低用户感知等待时间 |
| 复杂查询加载提示 | 必须有 | 跨模块查询、Web Search 等长链路场景，需提供友好的加载进度提示（如“正在检索企业数据库……”→“正在查询关联专利……”） |

### 5.2 意图识别与路由

| 指标 | 要求 | 说明 |
| --- | --- | --- |
| 意图识别准确率 | ≥ 90% | 用户指令意图清晰的常规场景 |
| 跨域路由错误率 | 尽可能低 | 如将“找论文”误判为“找企业”，这是用户体验的重大伤害点 |
| 模糊意图处理 | 追问澄清 | 意图不明确时进入多轮追问，不猜测 |

### 5.3 兼容性

| 要求 | 说明 |
| --- | --- |
| 接入形式 | 以 Web（H5）形式嵌入甲方微信公众号 |
| iOS 适配 | 通过 iOS 真机测试（含 Safari 内核 WebView），无样式错乱或交互卡顿 |
| Android 适配 | 通过 Android 真机测试（华为、小米、OPPO、vivo 等主流机型），无样式错乱或交互卡顿 |
| 响应式布局 | 适配主流手机屏幕（375px - 428px 宽度） |

### 5.4 平衡优化

在“意图识别精度”与“响应速度”之间寻找最佳平衡点。核心手段：

- **流式输出**：LLM 回答逐步输出，用户无需等待完整回答

- **异步加载**：先返回核心信息，关联数据（如教授卡片中的论文列表）后台加载后动态补充

- **加载交互提示**：复杂查询时展示进度，避免用户焦虑
